## 1. 시스템 목표 및 핵심 원칙

### 1.1 목표 (What)
MRA, AI_Music_Hub, Where_AI, Rental 등 **다중 로컬 프로젝트**의 기획·개발 산출물과, ChatGPT·Claude 등 **외부 상용 LLM 대화 내역**을 모두 취합하여 다음을 제공하는 **개인화 지식 검색 베이스**를 구축한다.

1. **지식의 중앙화**: 분산된 수만 건 문서·소스코드의 파편화된 컨텍스트를 하나의 검색 인터페이스로 연결
2. **시계열 이력 추적**: "이 문서의 초안 → 수정 → 최종본" 흐름과 "참고한 태스크노트 → 결과 보고서" 관계를 시각화
3. **API 비용 제로**: 기존 구독 중인 상용 LLM 웹 브라우저를 RPA로 연동하여 추가 API 비용 없이 활용
4. **로컬 독립형**: 기밀 문서 외부 유출 방지를 위해 클라우드 DB 배제, PC 내부에서 완결되는 Python 데스크톱 앱

### 1.2 원칙 (How — 제약 조건)

| 원칙 | 설명 |
|:---|:---|
| **데이터 주권** | 모든 인덱스·메타데이터·아카이브는 로컬 파일시스템에만 존재. 외부 전송 없음 |
| **원본 불변** | 프로젝트 원본 파일은 절대 복사·이동·수정하지 않음 (인덱스만 생성) |
| **점진적 복잡도** | Phase별로 최소 기능부터 구현. 각 Phase 완료 시 독립적으로 사용 가능한 상태 유지 |
| **실패 안전** | 파서·인덱싱·RPA 각 모듈은 실패해도 전체 시스템을 중단시키지 않음 (Graceful Degradation) |
| **감사 추적** | 모든 인덱싱·스크랩·RPA 실행은 로그 파일에 기록 (`logs/` 디렉토리) |

### 아키텍처 보완 포인트 (v1.0 대비)
- **SQLite 메타데이터 스토어 추가**: ChromaDB는 벡터 검색에 특화되어 있지만, 관계(Edge) 데이터·버전 체인·작업 로그 등 구조화된 메타데이터는 SQLite에 별도 관리

## 3. 데이터 보관 및 관리 정책

### 3.0 데이터 계약 (SSOT — Single Source of Truth)

모든 인덱싱·검색·그래프·버전 관리의 **기반이 되는 ID·스키마 규칙**을 먼저 확정합니다. 이 계약이 없으면 증분 인덱싱, 버전 체인, 검색 랭킹이 모두 불안정해집니다.

#### 3.0.1 ID 생성 규칙 (재현 가능·결정적)

| 대상 | ID 생성 공식 | 비고 |
|:---|:---|:---|
| **문서 (Document)** | `doc_id = sha256(정규화된 절대경로 + 파일크기 + mtime)` | 경로 정규화: 소문자 통일, 구분자 `/`로 통일, trailing slash 제거 |
| **청크 (Chunk)** | `chunk_id = sha256(doc_id + chunk_index + chunk_text_hash)` | 같은 문서 내에서 청크 순서와 내용이 모두 같아야 동일 ID |
| **엣지 (Edge)** | `edge_id = sha256(source_doc_id + target_doc_id + edge_type)` | 동일 쌍의 동일 유형 엣지는 1개만 유지 |
| **버전 체인** | `chain_id = sha256(정규화된 주제명)` | 주제명은 파일명에서 날짜 접두어 제거 후 정규화 |

> **핵심 원칙**: 동일한 입력이면 언제 실행해도 동일한 ID가 나와야 합니다 (결정적 생성). 이를 통해 중복 삽입, 인덱스 불일치 문제를 구조적으로 방지합니다.

#### 3.0.2 스키마 버전 관리

모든 레코드에 다음 3개 필드를 필수로 저장합니다:

| 필드 | 용도 | 예시 |
|:---|:---|:---|
| `schema_version` | DB 스키마 버전. 마이그레이션 판단 기준 | `1.0` |
| `indexer_version` | 인덱서 코드 버전. 재인덱싱 필요 여부 판단 | `0.2.1` |
| `embed_model_id` | 임베딩 모델 식별자. 모델 교체 시 전체 재임베딩 트리거 | `paraphrase-multilingual-MiniLM-L12-v2` |

> 임베딩 모델이 변경되면 기존 벡터와 신규 벡터의 차원/좌표계가 달라지므로, `embed_model_id` 불일치 감지 시 전체 재임베딩을 자동 제안합니다.

#### 3.0.3 원본 불변 조건의 기술적 강제

- 설정 파일에서 감시 경로를 **읽기 전용(read-only)**으로 등록
- 스캐너 모듈은 해당 경로에 대해 `os.open()` 시 `O_RDONLY` 플래그만 사용
- 쓰기 권한이 필요한 작업(아카이브 저장 등)은 `d:\Workspace_Brain\data\` 하위 경로로만 제한

### 3.1 내부 로컬 프로젝트: 인덱싱 전용 (Index-Only)

| 항목 | 세부 |
|:---|:---|
| **대상** | `d:\MRA`, `d:\AI_Music_Hub`, `d:\Where_AI`, `d:\Rental` |
| **지원 포맷** | `.md`, `.txt`, `.json`, `.ts`, `.js`, `.py`, `.html`, `.css` (설정으로 확장 가능) |
| **정책** | 원본 파일을 Workspace Brain 폴더로 복사하지 않음 |
| **인덱싱 방식** | 텍스트 추출 → 의미 단위 Chunk 분할 → 임베딩 벡터화 → ChromaDB 저장 |
| **메타데이터** | `doc_id`(§3.0), 절대 경로, 상대 경로, 파일명, 생성일, 수정일, 파일 크기, SHA-256 해시, 부모 프로젝트명 |
| **동기화** | 수동 버튼 + 선택적 File Watcher (자동) |
| **삭제 처리** | 원본 삭제 감지 시 인덱스에서 `soft-delete` 마킹 (물리 삭제가 아닌 비활성 처리) |

#### 증분 인덱싱 (Incremental Indexing)

**파일 단위가 아닌 청크 단위**로 증분 처리합니다:

1. 재동기화 시 파일의 SHA-256 해시를 비교하여 변경된 파일만 식별
2. 변경된 파일을 재파싱 → 새 청크 목록 생성
3. 기존 청크 vs 새 청크를 `chunk_id`로 비교
   - **동일**: 스킵 (DB 그대로 유지)
   - **변경/신규**: ChromaDB에 upsert + SQLite 메타 갱신
   - **소멸**: ChromaDB에서 soft-delete

#### rename/move 감지

| 상황 | 판단 기준 | 처리 |
|:---|:---|:---|
| 파일 이동 | 파일 해시 동일 + 경로 변경 | 동일 문서의 경로 업데이트 (버전 체인 유지) |
| 파일 이름 변경 | 파일 해시 동일 + 파일명 변경 | `doc_id` 재계산 + 이전 ID에서 신규 ID로 매핑 기록 |
| 파일 삭제 후 재생성 | 해시 다름 + 경로 동일 | 새 문서로 취급 (이전은 soft-delete) |

#### 파서 실패 격리 (_quarantine/)

파싱에 실패한 파일(인코딩 불일치, 바이너리 혼입, 파일 잠금 등)은 시스템을 중단시키지 않고 격리합니다:

- 실패 파일의 메타데이터(경로, 실패 사유, 타임스탬프)만 SQLite `quarantine_log`에 기록
- 물리적으로 파일을 이동하지 **않음** (원본 불변 원칙)
- UI의 "인덱싱 상태" 위젯에 "⚠️ 3건 파싱 실패" 표시 + 클릭 시 사유 목록 + [재시도] 버튼

#### 리컨실리에이션 (정합성 3자 교차 검증)

원본 파일 ↔ SQLite 메타 ↔ ChromaDB 청크 간 불일치를 정기적으로 탐지·복구합니다:

```
[원본 파일 시스템]   [SQLite 메타데이터]   [ChromaDB 벡터]
       ↕                    ↕                    ↕
   ① 파일 존재?         ② 메타 존재?        ③ 청크 존재?
       ↕                    ↕                    ↕
       └────────── 3자 비교 루프 ──────────────┘
                        ↓
              불일치 자동 수정 or 사용자 알림
```

| 불일치 유형 | 자동 조치 |
|:---|:---|
| 원본 있음 + 메타 없음 | 신규 인덱싱 실행 |
| 원본 없음 + 메타 있음 | soft-delete 마킹 |
| 메타 있음 + 청크 수 불일치 | 해당 문서만 재청킹·재임베딩 |
| 메타의 해시 ≠ 원본 해시 | 변경 감지 → 증분 인덱싱 |

- 실행 시점: 수동 버튼 + 선택적 일간 자동 실행 (설정 가능)
- 결과는 `logs/reconciliation_YYYYMMDD.log`에 기록

## 4. 시계열 추적 및 지식 그래프 (핵심 차별화)

### 4.1 타임라인 (Time-series) 버전 관리

```
[2026-02-28_설계초안.md] ──→ [2026-03-01_설계수정.md] ──→ [2026-03-03_설계최종.md]
         v1                           v2                           v3
```

**식별 로직**:
1. 파일명에서 `YYYY-MM-DD_` 접두어를 파싱 → 1차 시간 정렬
2. OS 레벨 생성일(ctime) / 수정일(mtime)을 2차 참조
3. 두 문서의 **파일명 유사도**(접두어 제거 후 비교) ≥ 70% AND 날짜 차이 ≤ 14일 → 동일 주제 후보
4. 후보 문서 간 **내용 코사인 유사도** ≥ 0.75 → 버전 체인으로 확정

**보완 사항 (v1.0 대비)**:
- v1.0은 "자동으로 묶는다"만 명시. 구체적 유사도 임계값과 판단 로직을 명시화
- 체인이 잘못 연결된 경우 사용자가 수동으로 해제/연결할 수 있는 UI 제공
- 버전 체인 메타데이터는 SQLite `version_chains` 테이블에 저장

### 4.2 지식 그래프 (Knowledge Graph)

문서를 **노드(Node)**, 문서 간 관계를 **엣지(Edge)**로 표현하는 방사형 네트워크.

#### 엣지 유형

| 유형 | 감지 방법 | 가중치 | 시각화 |
|:---|:---|:---:|:---|
| **명시적 링크** | 마크다운 내부 `[참조](파일명.md)` 형태 추출 | 1.0 | **굵은 실선** |
| **암묵적 유사** | 키워드 분포 + 임베딩 유사도 ≥ 0.7 AND 날짜 차 ≤ 7일 | 0.5~0.9 | **점선** (농도 = 유사도) |
| **버전 체인** | 4.1의 타임라인 로직으로 연결 | 1.0 | **화살표 실선** (시간 방향) |
| **프로젝트 동거** | 동일 프로젝트 폴더 내 하위 파일 | 0.3 | **연한 회색선** |

#### 보완: 그래프 확장성 및 성능 제한

| 제한 항목 | 기본값 | 설정 변경 |
|:---|:---|:---|
| 노드당 유사(암묵적) 엣지 | **최대 20개** | `config/settings.json`에서 조정 |
| 그래프 뷰 최대 표시 노드 | **200개** | 초과 시 클러스터 축약 뷰 자동 전환 |
| 커뮤니티 검출 임계값 | 노드 1,000개 초과 시 | Louvain 알고리즘으로 클러스터링 |

- 사용자가 관심 영역 줌인 시 해당 클러스터만 상세 전개
- 엣지 타입 필터를 기본 제공: 명시적 링크 / 버전 체인 / 유사도(임계값 슬라이더) / 프로젝트 동거
- 탐색 최소 동작: 노드 클릭 → 관련 문서 Top-10 → 미리보기 패널로 직접 연결

## 4.3 하이브리드 검색 랭킹 (Hybrid Search Ranking)

벡터 검색(의미 유사도)과 키워드 검색(정확 매칭)을 결합하여 검색 품질을 극대화합니다.
아키텍처(§2)에 "Hybrid: Vector + Keyword + Filter"가 명시되어 있으나, **정렬 규칙**이 없으면 결과 품질이 들쭉날쭉해집니다.

#### 검색 파이프라인

```
사용자 질의
    │
    ▼
┌──────────────────────────────────┐
│  ① 필터 선적용                   │  프로젝트/기간/확장자 조건으로
│     (메타데이터 기반)             │  후보 문서를 먼저 축소
└──────────────────────────────────┘
    │
    ├──────────┬──────────────────┐
    ▼          ▼                  │
┌────────┐ ┌────────────────┐     │
│ 벡터   │ │ 키워드 (FTS5)  │     │
│ Top-K  │ │ Top-K          │     │
└────────┘ └────────────────┘     │
    │          │                  │
    ▼          ▼                  │
┌──────────────────────────────────┐
│  ② RRF (Reciprocal Rank Fusion) │  두 결과 리스트를 합산 정렬
│     정렬 공식 적용               │
└──────────────────────────────────┘
    │
    ▼
  최종 결과 리스트
```

#### RRF 정렬 공식

```
RRF_score(d) = Σ  1 / (k + rank_i(d))
```

- `k` = 60 (상수, 랭킹 안정화용)
- `rank_i(d)` = i번째 검색 방식에서의 문서 d 순위 (1-index)

**계산 예시**:

| 문서 | 벡터 순위 (r_v) | 키워드 순위 (r_t) | 1/(60+r_v) | 1/(60+r_t) | RRF 합산 |
|:---|:---:|:---:|:---:|:---:|:---:|
| 문서 A | 3 | 10 | 0.01587 | 0.01429 | **0.03016** |
| 문서 B | 8 | 2 | 0.01471 | 0.01613 | **0.03084** |
| 문서 C | 1 | 50 | 0.01639 | 0.00909 | **0.02548** |

→ 최종 순서: **B > A > C** (벡터만 강한 C보다 양쪽 균형 잡힌 B가 상위)

#### 키워드 인덱스: SQLite FTS5

- 이미 SQLite를 메타 DB로 사용하므로 FTS5 가상 테이블 추가가 자연스러움
- 한국어는 `unicode61` 토크나이저 + 선택적 형태소 분석기(`mecab-python3`) 연동
- 벡터 검색이 약한 **정확한 고유명사/코드 식별자** 매칭을 보완

#### 랭킹 투명성 (UI)

검색 결과 카드에 "이 결과가 상위인 이유"를 투명하게 표시:

```
┌──────────────────────────────────────────┐
│  📄 2026-03-02_MRA_I18n_Analysis.md     │
│  MRA │ 보고서 │ 2026-03-02               │
│  "...I18n 적용 계획의 우선순위를..."     │
│  ─────────────────────────────────────── │
│  🔍 벡터 순위: #3  │  📝 키워드 순위: #1  │
│  RRF 점수: 0.0317                        │
└──────────────────────────────────────────┘
```

## 5. LLM 연동 전략 (통합 정리)

기존 3개 문서(`LLM_Integration_Options`, `LLM_No_API_Alternatives`, `LLM_Response_Scraping`)의 내용을 하나의 일관된 전략으로 통합합니다.

### 5.1 전략 개요: 3단계 LLM 활용 경로

```
┌─────────────────────────────────────────────────┐
│         LLM 활용 경로 (우선순위 순)              │
├─────────────────────────────────────────────────┤
│                                                 │
│  [경로 1] 로컬 임베딩 (필수/무료)               │
│  → 검색용. Sentence-Transformers 모델 사용      │
│  → 인터넷 불필요, GPU 선택적                    │
│                                                 │
│  [경로 2] RPA 연동 (핵심/무료)                  │
│  → 요약·추론용. 기존 구독 LLM 브라우저 활용     │
│  → 클립보드 매크로 + 핫키 스크랩                │
│                                                 │
│  [경로 3] 직접 API 호출 (선택/유료)             │
│  → 고급 RAG용. 설정 패널에서 선택적 활성화      │
│  → OpenAI/Gemini/Claude API 키 입력             │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 5.2 경로 1: 로컬 임베딩 (필수)

| 항목 | 세부 |
|:---|:---|
| **목적** | 문서 텍스트를 벡터로 변환하여 의미 기반 검색 수행 |
| **추천 모델** | `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` (384차원, 다국어, CPU 가능) |
| **대안** | 한국어 특화 필요 시 `jhgan/ko-sroberta-multitask` 고려 |
| **실행 환경** | CPU 전용 가능 (GPU 있으면 가속). 인터넷 불필요 (최초 다운로드 후) |
| **비용** | 완전 무료 |

> **v1.0 대비 보완**: "로컬 임베딩"이라고만 적혀 있던 부분을 구체적 모델명·크기·언어 적합성까지 명시.

### 5.3 경로 2: RPA 기반 상용 LLM 연동 (핵심 기능)

현재 구독 중인 ChatGPT Plus / Claude Pro 등의 **웹 브라우저 화면**을 직접 제어하여 API 비용 없이 고품질 LLM의 요약·추론 능력을 활용합니다.

#### 송신 (Workspace Brain → 브라우저)

1. 사용자가 검색 결과에서 관련 문서 3~5개를 선택
2. **`[🚀 ChatGPT로 질문 넘기기]`** 버튼 클릭
3. 프로그램이 지시 프롬프트 + 문서 내용을 하나의 텍스트로 조립
4. Windows 클립보드에 자동 복사
5. `win32gui` API로 브라우저 창을 포커스
6. `pyautogui`로 입력창에 `Ctrl+V` → `Enter` 자동 실행

#### 수신 (브라우저 → Workspace Brain)

1. 사용자가 LLM 답변을 브라우저에서 복사 (`Ctrl+C` 또는 복사 아이콘)
2. 약속된 단축키 `Ctrl+Shift+S` 입력
3. 백그라운드 핫키 리스너(`keyboard` 라이브러리)가 감지
4. 클립보드 내용을 `YYYY-MM-DD_LLM_스크랩_[자동요약].md`로 저장
5. 즉시 인덱싱 + 우하단 토스트 알림

#### 보완: RPA 안정성 대책

| 리스크 | 대책 |
|:---|:---|
| 브라우저 창을 못 찾음 | 창 제목 패턴 매칭 + 수동 창 지정 설정 UI |
| 입력창 위치 변경 | `pyautogui` 대신 `Ctrl+V` 단축키만 사용 (DOM 비의존) |
| 대용량 텍스트 붙여넣기 실패 | 15,000자 초과 시 "문서 리스트만 먼저" 전송 후, 사용자 요청 시 본문 추가 전송 (2단계 전략) |
| 핫키 충돌 | 설정 패널에서 사용자 정의 핫키 변경 가능 |
| 클립보드 오염 | 클립보드 보호 **기본값 ON**: 송신/수신 전후로 원래 클립보드 백업→복원 항상 수행 |
| 민감정보 노출 | 정규식 기반 자동 마스킹: API 키(`sk-...`), 토큰, 주민번호 패턴을 저장 전 `[MASKED]`로 치환. 로그·스크랩 모두 적용 |

### 5.4 경로 3: 직접 API 호출 (선택적 — 고급 RAG)

설정 패널에서 API 키를 입력하면 활성화되는 **완전 자동 RAG 파이프라인**입니다.
평소에는 경로 1(검색) + 경로 2(RPA)로 충분하지만, 다음과 같은 상황에서 API 직접 호출이 유용합니다.

- 검색 결과를 클릭 없이 **즉석에서 요약/비교 분석**하고 싶을 때
- RPA 조작 없이 **프로그램 내부에서 답변을 바로** 받고 싶을 때
- 대량 문서에 대한 **배치 요약/태깅 자동화**를 돌릴 때

#### 5.4.1 지원 프로바이더

| 프로바이더 | LangChain 패키지 / 클래스 | 추천 모델 (2026-03 기준) | 비고 |
|:---|:---|:---|:---|
| **OpenAI** | `langchain-openai` / `ChatOpenAI` | `gpt-5.2` (고급 추론), `gpt-5-mini` (가성비) | GPT-5 계열. Responses API 지원. 한국어 품질 최상 |
| **Google Gemini** | `langchain-google-genai` ≥4.0.0 / `ChatGoogleGenerativeAI` | `gemini-3.1-pro-preview` (최신 추론), `gemini-3-flash` (고속·가성비) | `google-genai` 통합 SDK 사용. Gemini 3 Pro는 2026-03-26 폐기 예정 → 3.1 Pro로 마이그레이션 |
| **Anthropic Claude** | `langchain-anthropic` / `ChatAnthropic` | `claude-sonnet-4.6` (밸런스), `claude-opus-4.6` (최강) | 1M 토큰 컨텍스트 윈도우 (beta). 2026-02 출시 |
| **로컬 Ollama** | `langchain-ollama` / `ChatOllama` | `llama3`, `mistral`, `gemma2` 등 | 완전 무료, GPU 권장. 오프라인 동작 |

> **설계 원칙**: LangChain 1.0 LTS의 `BaseChatModel` 인터페이스로 추상화하므로, 프로바이더 추가 시 설정 파일에 클래스명·모델명만 등록하면 코드 수정 없이 확장 가능.
>
> **SDK 참고**: `langchain-google-genai` 4.0.0부터 Google의 통합 `google-genai` SDK를 사용하며, Gemini Developer API와 Vertex AI 양쪽 백엔드를 단일 인터페이스로 지원합니다. OpenAI는 기존 Chat Completions API 외에 Responses API도 지원하여 GPT-5의 전체 기능(자유형 함수 호출, 상태 관리 등)에 접근할 수 있습니다.

#### 5.4.2 RAG 파이프라인 흐름

```
사용자 질의
    │
    ▼
┌─────────────────────────────────┐
│  1. 벡터 검색 (ChromaDB)        │  ← 경로 1 (로컬 임베딩)
│     → Top-K 관련 문서 추출      │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│  2. 컨텍스트 조립               │
│     - 시스템 프롬프트 (역할 지시)│
│     - 검색된 문서 Chunk 삽입    │
│     - 사용자 원문 질의 부착     │
│     - 토큰 상한 체크 & 자동 축소│
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│  3. API 호출                    │  ← 경로 3 활성화 시
│     - 선택된 프로바이더로 전송  │
│     - 스트리밍 응답 수신        │
│     - 토큰 사용량 기록          │
└─────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────┐
│  4. 응답 표시 & 저장            │
│     - UI 인사이트 패널에 렌더링 │
│     - 선택적: .md로 아카이브    │
│     - 출처 문서 링크 함께 표시  │
└─────────────────────────────────┘
```

#### 5.4.3 API 키 관리 (보안)

| 항목 | 설계 |
|:---|:---|
| **저장 방식** | `settings.json`에 평문 저장하지 **않음**. OS 자격 증명 관리자(`keyring` 라이브러리) 사용 |
| **Windows** | Windows Credential Manager에 암호화 저장 |
| **입력 UI** | 설정 패널의 비밀번호 마스킹 입력 필드 (`●●●●●●`) |

| **삭제** | 설정에서 "키 삭제" 버튼 → 자격 증명 관리자에서 즉시 제거 |

### 5.5 경로 간 폴백(Fallback) 전략

세 가지 경로가 공존하는 상황에서의 우선순위와 자동 전환 로직:

```
사용자가 "AI 요약" 버튼 클릭
    │
    ├── API 키 등록됨? ──YES──→ 경로 3 (API 직접 호출)
    │                               │
    │                          호출 실패? (키 만료/상한 초과/네트워크)
    │                               │
    │                              YES
    │                               │
    │                               ▼
    ├── NO ───────────────────→ 경로 2 (RPA) 시도
    │                               │
    │                          브라우저 창 감지 실패?
    │                               │
    │                              YES
    │                               ▼
    └────────────────────────→ 안내 메시지 표시:
                               "API 키를 등록하거나,
                                ChatGPT 브라우저를 열어주세요"
```

| 상황 | 자동 행동 |
|:---|:---|
| API 키 유효 + 상한 이내 | 경로 3으로 즉시 호출 |
| API 키 유효 + 상한 초과 | 경고 표시 + 경로 2로 폴백 |
| API 키 미등록 | 경로 2 자동 시도 |
| API 실패 (네트워크 등) | 에러 표시 + 경로 2로 폴백 |
| 경로 2도 실패 (브라우저 없음) | 검색 결과만 표시 + 안내 메시지 |

> **v1.0 대비 보완**: 기존에 "옵션 A/B/C"로 나열만 되어 있던 3가지 방식을 **우선순위가 있는 단일 전략**으로 통합. "경로 1은 필수, 경로 2가 핵심, 경로 3은 선택"이라는 명확한 위계를 설정하고, API 설계(프로바이더 지원, RAG 파이프라인, 키 보안, 비용 관리, 폴백 전략)를 구체화.

## 10. 보안 고려사항

| 영역 | 위협 | 대책 |
|:---|:---|:---|
| **데이터 저장** | ChromaDB/SQLite가 평문 저장 | 로컬 전용이므로 OS 수준 디스크 암호화(BitLocker) 권장. 앱 수준 암호화는 Phase 5에서 검토 |
| **클립보드** | RPA 송신 시 민감 문서가 클립보드에 잔류 | 송신 완료 후 클립보드 즉시 초기화 (기본값 ON) |

| **민감정보** | 스크랩/로그에 API 키·토큰·개인정보가 포함 | 정규식 기반 자동 마스킹 룰: `sk-*`, `Bearer *`, 주민번호 패턴 등을 저장 전 `[MASKED]`로 치환 |

## 11. 테스트 전략

### 11.1 테스트 유형

| 수준 | 대상 | 도구 |
|:---|:---|:---|
| **단위 테스트** | 파서, 청킹, 날짜 추출, 해시 계산, 유사도 판정, ID 생성(§3.0) | pytest |
| **통합 테스트** | 스캔 → 인덱싱 → 검색 → 결과 반환 E2E | pytest + 임시 폴더 fixture |
| **RPA 테스트** | 클립보드 복사/붙여넣기, 핫키 등록/해제 | 수동 + pyautogui 시뮬레이션 |
| **성능 테스트** | 10,000건 문서 인덱싱 시간, 검색 응답 시간 | pytest-benchmark |
| **UI 테스트** | Streamlit 렌더링, 위젯 상호작용 | Streamlit 테스트 유틸 + 수동 |

### 11.2 필수 품질 게이트 (DoD: Definition of Done)

각 Phase 완료 시 아래 3종 테스트를 통과해야 "완료"로 인정합니다:

| 테스트 | 내용 | 실패 시 |
|:---|:---|:---|
| **파서 골든 테스트** | 샘플 파일 N개를 파싱한 결과가 사전 정의된 기대 출력과 100% 일치 | 파서 수정 후 재통과 |
| **인덱스 무결성 테스트** | 원본 파일 수 = SQLite 메타 수 = Chroma 문서 수 (리컨실리에이션 §3.1 기반) | 불일치 사유 조사 후 수정 |
| **검색 리그레션 테스트** | 고정 질의 목록(예: 30개)에 대해 Top-3 결과가 이전 빌드 대비 급변(50% 이상 교체) 시 경고 | 랭킹 로직 변경 사유 확인 |

### 11.3 DB 백업 및 복구

로컬 독립형 프로그램이므로 사용자의 PC 사고(디스크 손상, 실수 삭제)에 대비한 백업이 필수입니다.

| 항목 | 설계 |
|:---|:---|
| **대상** | SQLite `metadata.db` + ChromaDB `chroma_db/` 디렉토리 |
| **방식** | `data/backups/YYYYMMDD_HHMMSS/`에 전체 스냅샷 복사 |
| **주기** | 인덱싱 실행 전 자동 백업 (설정 변경 가능) |
| **보관** | 최근 7개 스냅샷 유지, 초과분 자동 정리 (FIFO) |
| **복구** | 설정 패널에서 "백업에서 복원" 버튼 → 스냅샷 선택 → 덮어쓰기 |
| **인덱싱 중단-재시작** | 배치 처리 시 1,000건마다 체크포인트 저장. 비정상 종료 후 재실행 시 마지막 성공 지점부터 자동 이어서 진행 |
