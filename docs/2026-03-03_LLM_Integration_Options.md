# LLM 연동 방식 상세 설명

## 1. 프로그램이 직접 LLM에 질의 및 전달 가능한가?

**네, 완벽하게 가능합니다.** 이 부분이 바로 구축하려는 시스템(RAG 기반 지식 베이스)의 핵심입니다.

프로그램은 단순히 텍스트를 검색하는 것에 그치지 않고, 다음의 과정을 자동으로 수행합니다.
1. 사용자가 질문을 입력합니다. (예: "MRA 프로젝트의 I18n 적용 계획이 뭐야?")
2. 시스템이 내부 데이터베이스에서 가장 연관성 높은 사내 문서(Markdown, 텍스트 등) 3~5개를 찾아냅니다.
3. 시스템이 찾아낸 **문서 내용(컨텍스트)을 LLM에 전송**하면서 "이 사내 문서를 바탕으로 사용자 질문에 답해줘"라고 백그라운드에서 직접 질의합니다.
4. LLM이 생성한 정리된 답변을 받아서 UI에 보여줍니다.

이 모든 과정이 프로그램 내부 코드(예: LangChain 라이브러리)를 통해 자동화되어 처리됩니다.

---

## 2. 로컬 LLM이나 상용 LLM API를 쓸 수밖에 없는가?

**현실적으로 그렇습니다.** 프로그램이 텍스트의 "의미"를 이해하고 자연어 요약을 생성하려면 추론 능력을 가진 인공지능(LLM) 모델이 반드시 필요하기 때문입니다. 

하지만 어떤 형태의 LLM을 선택할지는 완벽한 통제권을 가질 수 있습니다. 크게 세 가지 선택지가 있습니다.

### 옵션 A: 상용 클라우드 API 사용 (가장 똑똑하고 편리함)
*   **방식**: OpenAI (ChatGPT), Google Cloud (Gemini), Anthropic (Claude) 등의 API 키를 발급받아 프로그램에 입력합니다.
*   **장점**: 답변 수준이 압도적으로 높고, 복잡한 문서 요약이나 추론에 가장 능합니다. 개발 및 연동이 매우 쉽습니다.
*   **단점**: 사용량에 따른 API 비용이 발생하며, 회사 내부 기밀 문서가 외부 클라우드 서버 망을 타고 전송됩니다.

### 옵션 B: 로컬 LLM 사용 (무료 및 보안 100%)
*   **방식**: 사용자의 PC(또는 사내 서버)에 직접 LLM 모델을 다운로드하여 실행합니다. (Ollama, LM Studio 등을 활용)
*   **장점**: 내부 문서가 외부 인터넷으로 절대 나가지 않아 보안이 완벽합니다. 인터넷 연결 없이도 오프라인 환경에서 작동하며 평생 무료입니다.
*   **단점**: 사용자 PC의 그래픽카드(GPU) 성능에 크게 의존합니다. 속도가 좀 느리거나 상용 API에 비해 답변의 똑똑함이 다소 떨어질 수 있습니다.

### 옵션 C: 하이브리드 방식 (추천)
*   단순한 키워드 기반 **일반 검색(Elasticsearch 류) + 벡터 검색 구조**까지만 자체적으로 구현하고, LLM 질의는 사용자가 버튼을 눌렀을 때만 선택적으로 동작하게 만듭니다. 
*   평소에는 "내부 문서 검색 및 관계 그래프 보기" 도구로만 가볍게 쓰고, 요약이 필요할 때만 상용 API를 소량 호출하여 비용과 성능의 밸런스를 잡습니다.

**결론적으로, 두뇌 역할을 하는 LLM 엔진 자체를 밑바닥부터 새로 개발할 수 없으므로(천문학적 비용 발생), 이미 만들어진 로컬 모델을 내 PC에 다운로드해서 쓰거나(무료/보안), 상용 API를 연결해서 쓰는(유료/고성능) 융통성 있는 구조를 선택하는 것이 표준입니다.**
