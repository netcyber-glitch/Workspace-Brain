# 데이터 저장 방식: 원본 유지 vs 인덱스 수집

이 질문은 매우 중요한 시스템 설계 포인트입니다. 엄청난 용량의 프로젝트 코드를 매번 복사하면 디스크가 낭비되고 동기화 문제가 발생할 수 있습니다.

**결론부터 말씀드리면, "원본 파일은 기존 프로젝트 그대로 두고, 내용의 인덱스(Index) 기반 벡터(숫자)와 메타데이터만 수집" 하는 것이 정석입니다.**

하지만 문서의 성격에 따라 처리 방식이 약간 다릅니다.

---

## 1. MRA, AI_Music_Hub 등 "로컬 프로젝트" 내의 파일들
- **처리 방식**: **인덱스와 메타데이터만 수집 (원본 유지)**
- **동작 원리**:
  1. 프로그램이 `d:\MRA`, `d:\AI_Music_Hub` 등의 폴더를 스캔합니다.
  2. 원본 문서를 통째로 복사해서 `Workspace_Brain` 폴더로 옮기지 **않습니다**.
  3. 대신 원본 문서의 내용 텍스트를 AI가 이해할 수 있는 형태(Vector)로 잘라서 **로컬 데이터베이스(ChromaDB 등)** 에 저장하고, 원본 파일이 있는 "절대 경로(`d:\MRA\docs\reports\...`)"만 기억합니다.
  4. 사용자가 검색하면, DB에서 해당 경로를 꺼내어 "이 파일에 질문에 대한 답이 있습니다" 라고 알려주고, 클릭 시 즉각 원본 파일을 열어줍니다.
- **장점**: 용량 낭비가 없고, MRA에서 파일을 수정하면 Workspace Brain에서 [초기화/동기화] 버튼 한 번만 누르면 최신 내용으로 인덱스만 조용히 덮어씁니다.

## 2. ChatGPT Export, 외부 수집 등 "출처가 없는" 파일들
- **처리 방식**: **전체 데이터 물리적 저장 (보관소 역할)**
- **동작 원리**:
  1. ChatGPT 메뉴에서 다운받은 `conversations.json` 같은 아카이브 데이터, 또는 사용자가 브라우저에서 스크랩(복사-복붙 핫키)한 답변 텍스트들은 돌아갈 '원본 폴더'가 마땅치 않습니다.
  2. 이런 파편화된 외부 텍스트 지식은 `d:\Workspace_Brain\data\archive` 같은 **전용 보물창고 폴더**에 물리적인 `.md` 파일 형태로 영구 저장시킵니다.
  3. 당연히 저장된 직후 이 파일들도 1번의 방식과 똑같이 벡터DB에 인덱싱되어 검색망에 편입됩니다.
- **장점**: 브라우저나 외부 서비스가 날아가도 내가 모은 지식은 평생 내 하드디스크에 파일 형태로 남습니다.

---

## 핵심 요약 (Storage Policy)

1. **내 프로젝트(개발/기획) 문서들**: 절대 복사하지 않고, 주소와 내용 지문(Index)만 떠갑니다. 도서관의 **"도서 검색 카드(목록)"**만 만드는 것과 같습니다.
2. **외부 크롤링/스크랩 답변들**: Workspace Brain이 **"원본 저장소"** 역할까지 겸하며 물리적으로 파일(`.md`)을 찍어냅니다. 도서관에 아예 **"새 책을 사서 들여놓는 것"**과 같습니다.

이렇게 역할을 분리해야 수백 기가바이트의 소스코드나 문서들이 중복 복사되는 끔찍한 사태(결국 용량 터짐)를 막을 수 있고, 검색 속도는 로컬 DB의 힘을 빌려 0.1초 만에 튀어나오게 됩니다.
